## Purpose

This document pre-commits the specific hypotheses that the first data
window of Ambient Chronomancy will be used to test. It is signed by
this git commit and timestamped by GitHub. Any analysis that deviates
from what is written here is either a labeled extension or misconduct.

## Operational definitions (frozen for this prereg)

- **Ambient event** — a 60-second collector window with `p_combined < 0.01` (ledger-fire threshold).
- **Ping-eligible event** — an ambient event with `p_combined < 0.001`.
- **Dispatched ping** — a ping-eligible event for which a Telegram message was delivered to a user whose current ambient_mode is `ambient_only` or `both_50_50`, subject to per-user rate limit of 3/day.
- **Responded** — a user provided an UP, DOWN, or tag response to a dispatched ping within 30 minutes of delivery.
- **Ignored** — no response within 30 minutes.
- **UP-rate-given-response** — `count(UP) / (count(UP) + count(DOWN))` excluding tag-only and ignored pings.
- **Detector magnitude** — `−log₁₀(p_combined)` at the moment of the ping.

## Primary hypothesis — H3 (within ambient mode)

**H3.** Among pings dispatched while a user is in `ambient_only` or
`both_50_50` mode *and* that user responded, pings with higher detector
magnitude will have higher UP-rate-given-response.

- **Test:** Spearman rank correlation between detector magnitude and UP=1/DOWN=0 indicator, over all responded pings in the data window.
- **Null:** ρ = 0.
- **Alternative:** ρ > 0 (one-sided).
- **Pre-registered α:** 0.05.
- **Minimum N:** analysis runs when N_responded ≥ 200 responded pings across all ambient-mode users, or at the 100-day mark, whichever comes first.
- **Interpretation:**
  - ρ > 0 and p < 0.05 → directional support for MPI framing (detector-magnitude couples to user's self-reported meaningfulness).
  - ρ ≤ 0 or p ≥ 0.05 → no evidence against null; instrument does not show the coupling its design is predicated on within this data window.

## Secondary hypothesis — H4 (between modes, concurrent A/B)

**H4.** For users whose ambient_mode is `both_50_50` (concurrent ambient
+ standard pings within the same window, mode randomized per dispatched
ping, mode blinded until response), the UP-rate-given-response will be
higher for ambient-sourced pings than for standard-sourced pings.

- **Test:** two-proportion z-test, one-sided (ambient > standard).
- **Null:** P_UP_ambient = P_UP_standard.
- **Alternative:** P_UP_ambient > P_UP_standard.
- **Pre-registered α:** 0.05.
- **Minimum N:** runs when each arm has ≥ 100 responded pings within-subject.
- **Blinding:** the Telegram ping text is identical across modes; source mode recorded in `ambient_pings` but not shown to the user in the ping itself. User can query historical mode breakdown after responding via `/ambient history`.
- **Interpretation:**
  - significant difference in the predicted direction → directional support for MPI framing over DAT framing (the embedding of the stream in meaningful context, not the scheduled-decision moment, is the channel).
  - not significant → no within-subject distinction; consistent with DAT, consistent with null phenomenon, or consistent with instrument underpowered.

## Tertiary hypotheses — sidereal and geomagnetic (H1, H2)

H1 and H2 from AMBIENT_GAMIFICATION.md §"Pre-registration format" are
carried over unchanged. Briefly: the distribution of ambient events over
local sidereal time (H1) and its correlation with the Kp geomagnetic
index (H2). These operate on all ledger-fire ambient events (no
response filter) and so are tested on aggregate log data.

- **H1 test:** Rayleigh test for non-uniformity of event-time circular distribution on 24h sidereal clock.
- **H2 test:** Spearman correlation of daily aggregate event rate vs daily Kp index, one-sided positive.
- **Minimum N:** 10,000 ledger-fire events or 100 days, whichever first.

## Family-wise error control

Four tests (H1, H2, H3, H4) with Bonferroni: α_per_test = 0.0125 for
family significance. Individual nominal α of 0.05 is retained for
reporting, but a "family-significant" claim requires at least one test
at p < 0.0125.

## Data window and locking

- **Window opens:** on the first dispatched ping after this
  pre-registration commit lands. Timestamp will be recorded as the
  commit time of the first `daily/` ledger entry following this file.
- **Window closes:** at 100 calendar days from window open, or at
  10,000 ledger-fire ambient events, or 200 responded pings (whichever
  H3 needs first). Whichever terminates first.
- **Analysis runs:** within 30 days of window close. Results committed
  to `results/` in this repo with a link back to this pre-registration
  file.
- **No data peeking before close.** The aggregate analysis is not
  computed until the window terminates. Per-moment operator-facing
  displays (p-value attached to each past ping) continue to be shown
  to the operator — that is part of the instrument, not analysis.

## What this pre-registration does not do

- It does not test whether single-operator data is sufficient to
  decisively resolve MPI vs DAT (it isn't). The pilot phase runs with
  the author as the only user. The N thresholds above are set so that
  H3 may reach power with a single operator; H4 requires a second user
  opted into `both_50_50` and contributing responses.
- It does not commit to any post-hoc extensions. If the data window
  produces surprising patterns not anticipated here, they will be
  described as exploratory extensions in a separate commit, not
  retrofitted into this file.
- It does not promise any particular result. The instrument may find
  nothing, and the ledger will honestly say so.
